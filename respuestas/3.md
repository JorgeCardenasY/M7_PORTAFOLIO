# Implementación de Modelos con Relaciones en Django

Para abordar la problemática de modelar entidades interconectadas utilizando los tipos de relaciones proporcionados por Django, se han creado tres modelos principales: `Cliente`, `Producto` y `Pedido`. Estos modelos demuestran las relaciones de uno a muchos (`ForeignKey`) y de muchos a muchos (`ManyToManyField`).

## Modelos Implementados

### 1. `Cliente`

Este modelo representa a un cliente. Es una entidad simple con campos básicos para identificar a cada cliente.

```python
class Cliente(models.Model):
    nombre = models.CharField(max_length=100)
    apellido = models.CharField(max_length=100)
    email = models.EmailField(unique=True)

    def __str__(self):
        return f"{self.nombre} {self.apellido}"
```

### 2. `Producto`

Este modelo representa un producto que puede ser parte de un pedido. También es una entidad simple con información básica del producto.

```python
class Producto(models.Model):
    nombre = models.CharField(max_length=100)
    descripcion = models.TextField(blank=True)
    precio = models.DecimalField(max_digits=10, decimal_places=2)

    def __str__(self):
        return self.nombre
```

### 3. `Pedido`

Este es el modelo central que establece las relaciones con `Cliente` y `Producto`.

```python
class Pedido(models.Model):
    cliente = models.ForeignKey(Cliente, on_delete=models.CASCADE, related_name='pedidos')
    productos = models.ManyToManyField(Producto, related_name='pedidos')
    fecha_pedido = models.DateTimeField(auto_now_add=True)
    total = models.DecimalField(max_digits=10, decimal_places=2, default=0.00)

    def __str__(self):
        return f"Pedido #{self.id} de {self.cliente.nombre}"

    def save(self, *args, **kwargs):
        super().save(*args, **kwargs)
        if not self.total and self.productos.exists():
            self.total = sum(producto.precio for producto in self.productos.all())
            super().save(update_fields=['total'])
```

## Explicación de las Relaciones

### Relación Uno a Muchos (`ForeignKey`): `Cliente` y `Pedido`

*   **Objetivo:** Un cliente puede realizar múltiples pedidos, pero cada pedido pertenece a un único cliente.
*   **Implementación:** En el modelo `Pedido`, hemos definido el campo `cliente` como un `ForeignKey` que apunta al modelo `Cliente`.
    ```python
    cliente = models.ForeignKey(Cliente, on_delete=models.CASCADE, related_name='pedidos')
    ```
    *   `Cliente`: Es el modelo al que se está relacionando.
    *   `on_delete=models.CASCADE`: Indica que si un `Cliente` es eliminado, todos sus objetos `Pedido` asociados también serán eliminados. Es relevante mencionar que se esto se implementa para mantener la integridad referencial de la BBDD.
    *   `related_name='pedidos'`: Permite acceder a los pedidos de un cliente desde una instancia de `Cliente` (ej. `mi_cliente.pedidos.all()`).

### Relación Muchos a Muchos (`ManyToManyField`): `Producto` y `Pedido`

*   **Objetivo:** Un pedido puede contener múltiples productos, y un mismo producto puede aparecer en múltiples pedidos.
*   **Implementación:** En el modelo `Pedido`, hemos definido el campo `productos` como un `ManyToManyField` que apunta al modelo `Producto`.
    ```python
    productos = models.ManyToManyField(Producto, related_name='pedidos')
    ```
    *   `Producto`: Es el modelo con el que se establece la relación muchos a muchos.
    *   `related_name='pedidos'`: Permite acceder a los pedidos en los que aparece un producto desde una instancia de `Producto` (ej. `mi_producto.pedidos.all()`).

## Funcionamiento Adicional en `Pedido`

Se ha incluido un método `save` sobrescrito en el modelo `Pedido` para calcular automáticamente el `total` del pedido. Este es un ejemplo simple de cómo se puede añadir lógica de negocio directamente en los modelos. Es importante notar que para que el `total` se calcule correctamente, los productos deben ser añadidos al pedido *antes* de que se guarde el pedido por segunda vez (o se actualice) para que el `save` pueda _iterar sobre_ los productos en `self.productos.all()`.

## Conclusión

En conclusión, se puede observar que estos modelos demuestran cómo Django facilita la creación de esquemas de base de datos complejos y relacionados utilizando su ORM, permitiendonos a los desarrolladores trabajar con objetos Python en lugar de SQL directo para gestionar las interacciones entre diferentes entidades de datos.
